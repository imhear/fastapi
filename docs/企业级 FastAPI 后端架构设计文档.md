## 二、分层职责说明（宣讲专用）

### 1. 接口层（Router）
- 负责：HTTP 请求接收、参数校验、响应格式统一
- 注入：`DB Session` + `Composer`
- 原则：**只做 HTTP 适配，不写业务逻辑**

### 2. 组合层（Composer）
- 定位：**跨模块业务编排中心**
- 作用：将用户、角色、权限等多个服务串联
- 特点：不管理事务、不创建会话，只做流程组装

### 3. 服务层（Service）
- 定位：**单一模块业务核心**
- 职责：用户模块 / 角色模块 / 权限模块 内聚逻辑
- 特点：纯业务、无框架耦合、可独立测试

### 4. 仓储层（Repository）
- 定位：**数据访问层**
- 职责：SQL 拼装、模型操作
- 规则：**必须接收外部传入 Session，不自建连接**

### 5. 会话管理层（DB Session）
- 唯一入口：`get_db`
- 管理：请求级生命周期、自动提交/回滚
- 保证：**跨模块天然原子事务**

### 6. 依赖注入容器（DI Container）
- 作用：统一管理 Service / Repository 实例
- 原则：**只做依赖创建，不管理数据库会话**

# 三、核心架构亮点（可直接宣讲）

## 1. 事务统一管理，无分布式事务问题
- 整个系统 **唯一事务入口：get_db**
- 一次请求 → 一个 Session → 贯穿所有模块
- **更新失败自动全回滚，数据强一致**

## 2. 贴合 FastAPI 原生生态，无过度设计
- 不自定义 UoW、不重复封装事务
- 符合框架设计哲学，**性能最优、兼容性最强**

## 3. 真正的高内聚、低耦合
- 模块边界清晰：User / Role / Permission 完全隔离
- 业务逻辑不与框架耦合，**可迁移、可替换、可测试**

## 4. 可维护性拉满
- 新人 10 分钟可上手
- 所有依赖关系在一个容器中可见
- 事务问题、数据问题**定位极快**

## 5. 企业级可扩展性
- 可轻松扩展：Redis、MQ、第三方服务、文件服务
- 可直接接入：登录认证、权限中心、操作日志、监控告警

# 五、关键架构决策（非常重要，可直接讲）

## 决策 1：采用 FastAPI 原生 Depends 管理 DB Session
- **放弃**：自定义工作单元（UoW）
- **放弃**：容器管理数据库连接
- **理由**：
  - 框架原生 > 自定义封装
  - 事务统一、代码极简、无坑
  - 大厂中后台标准实践

## 决策 2：使用单根 DI 容器，不做复杂父子容器
- **保留**：依赖注入、模块化隔离
- **放弃**：多层父子容器
- **理由**：
  - FastAPI 单请求模型不需要多级作用域
  - 简化结构、降低调试成本
  - 业务系统不需要插件化架构

## 决策 3：仓储层只接收外部 Session，不自建连接
- **好处**：
  - 天然支持跨模块事务
  - 同一请求内读写一致
  - 解决“查询返回旧数据”经典问题

## 决策 4：业务分层严格隔离
- Router：不写业务
- Service：单一模块职责
- Composer：跨模块编排
- Repository：只做数据访问

# 六、对比旧架构（优势总结）

| 维度 | 旧架构（父子容器 + 自建 UoW） | 新架构（FastAPI 原生 + 单容器） |
|------|------------------------------|--------------------------------|
| 事务管理 | 复杂、多入口、易出错 | 统一入口、自动提交回滚 |
| 代码量 | 多 30% | 极简、无冗余 |
| 学习成本 | 高（需懂容器+UoW） | 低（原生 FastAPI 即可） |
| 数据一致性 | 易出现旧数据查询 | 同 Session 强一致 |
| 可维护性 | 一般 | 极高 |
| 生态适配 | 一般 | 完美 |
